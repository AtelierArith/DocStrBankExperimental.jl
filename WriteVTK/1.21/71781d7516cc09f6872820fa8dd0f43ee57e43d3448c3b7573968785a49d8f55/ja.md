```
pvtk_grid(
    filename, args...;
    part, nparts, extents, ismain = (part == 1), ghost_level = 0,
    kwargs...,
)
```

並列 VTK ファイルを表すハンドラを返します。これは最終的に [`close`](@ref) を使ってファイルに書き込むことができます。

`args` と `kwargs` の位置引数およびキーワード引数は、[`vtk_grid`](@ref) にそのまま渡されます。シリアルファイル名は、`filename` とプロセス ID `part` から自動的に生成されることに注意してください。

以下のキーワード引数は、並列 VTK ファイル形式にのみ適用されます。

必須の引数は次のとおりです：

  * `part`: 現在の (1 ベースの) 部分 ID、
  * `nparts`: 部分の総数 ( **非構造** グリッドにのみ必要)、
  * `extents`: 異なるプロセス間での **構造** グリッドのパーティショニングを指定する配列 (詳細は以下を参照)。

オプションの引数は次のとおりです：

  * `ismain`: 現在の部分 ID `part` がメインである場合は `true` (唯一の `.pvtk` ファイルを書き込む部分)、
  * `ghost_level`: ゴーストレベル。

## 構造グリッドのためのエクステントの指定

構造グリッドの場合、データセットの異なるプロセス間でのパーティショニングは `extents` 引数を介して指定する必要があります。これは、各要素が特定のプロセスに関連付けられたデータのエクステントを表す配列です。

たとえば、4 つのプロセスに分散されたグローバル次元 $15×12×4$ のデータセットの場合、この配列は次のようになります：

```julia
extents = [
    ( 1:10,  1:5, 1:4),  # プロセス 1
    (10:15,  1:5, 1:4),  # プロセス 2
    ( 1:10, 5:12, 1:4),  # プロセス 3
    (10:15, 5:12, 1:4),  # プロセス 4
]
```

いくつかの重要な注意点：

  * `extents` 引数はすべてのプロセスで同じでなければなりません；
  * エクステントは **重複している必要があります**。そうしないと、VTK / ParaView はファイルを開こうとしたときに不満を言います；
  * `extents` 配列の長さはプロセスの数を示します。したがって、`nparts` 引数は冗長であり、渡す必要はありません。
