[`DafReader`](@ref) インターフェースは、`Daf` データを読み取るための高レベル API を指定します。この API は、低レベルの [`FormatReader`](@ref) API の上に実装されています。高レベル API はスレッドセーフを提供するため、低レベル API はこの問題を（ほとんど）無視できます。

各データセットには、エラーメッセージなどで使用するための名前が付けられます。この名前は、`Daf` オブジェクトを作成する際に明示的に設定できます。そうでない場合、既存のデータセットを開くときに、スカラー "name" プロパティが含まれていれば、それが使用されます。そうでなければ、いくつかの合理的なデフォルトが使用されます。すべての場合において、オブジェクト名は曖昧さを避けるために [`unique_name`](@ref) を通じて渡されます。

データプロパティは、それに基づく軸によって与えられたユニークな名前で識別されます。つまり、スカラープロパティ、特定の軸ごとのベクタープロパティ、および各 **無秩序** 軸のペアごとのマトリックスプロパティには別々の名前空間があります。

マトリックスについては、その [`MatrixLayouts`](@ref) を注意深く追跡します。返されるマトリックスは常に列優先レイアウトであり、必要に応じて [`relayout!`](@ref) を使用します。この操作は高コストであるため、結果をメモリにキャッシュします。同様に、データにクエリを適用した結果もキャッシュします。必要に応じてメモリ使用量を減らすためにキャッシュをクリアすることも許可します。

データ API は、パッケージの外部から使用されることを意図した高レベル API であり、したがってトップレベルの `Daf` 名前空間から再エクスポートされています。これは、低レベルの [`FormatReader`](@ref) 実装の上に追加機能を提供し、より一般的なデータ型を受け入れ、必要に応じて自動的に [`relayout!`](@ref) に対処します。特に、各データセットに対して単一のライターと複数のリーダーを強制するため、フォーマットコードはマルチスレッド処理を無視でき、依然としてスレッドセーフです。

!!! note
    以下の API では、値を取得する際に `default` を `undef` に指定することは、値が存在しないことが `error` であることを意味します。対照的に、`default` を `nothing` に指定することは、値が存在しないことが OK であり、`nothing` を返すことを意味します。`default` に実際の値を指定することは、値が存在しないことが OK であり、代わりに `default` を返すことを意味します。これは、"初期化子がない" と言う配列構築のフラグとして `undef` が使用されることに精神的には一致しますが、同一ではありません。これが `undef` 値の乱用であると感じる場合、`default` のデフォルト値であるため、コードに明示的に書く必要がほとんどないことに安心してください。
